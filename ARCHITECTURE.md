# Architecture: Local VLM Image Tagger Benchmark

## 1. Purpose and Scope

This project is a local, CLI-first benchmark runner for vision-language models served by LM Studio.

It is intentionally simple:
- single-process execution;
- deterministic file artifacts;
- no web backend, no database, no async scheduler.

The architecture describes **technical contracts** and component boundaries.
Workflow and user-facing usage details belong in `PROJECT_GUIDE.md`.

## 2. Non-goals

Out of scope for v1:
- GUI;
- web server;
- SQLite or other persistent DB;
- distributed execution;
- automatic quality judging model;
- non-LM-Studio model backends.

## 3. Runtime Pipeline

Canonical run path (`main.py run`):

1. Load config (`src/config.py`).
2. Detect config style:
   - full internal config;
   - simple user profile (`config.yaml`) expanded via `src/user_config.py`.
3. Validate semantic constraints (`src/validator.py`).
4. Load pools (`src/tag_pools.py`).
5. Discover input images (`src/image_loader.py`).
6. Create run directory and manifest artifacts (`src/storage.py`).
7. Connect to LM Studio (`src/lmstudio_client.py`).
8. For each selected model:
   - unload leftovers (best effort);
   - load model with configured load payload;
   - optional image smoke request;
   - run requests for each `(image, mode)`;
   - save raw + normalized artifacts;
   - append `summary.csv` row per request;
   - unload model.
9. Build reports (`src/report.py`):
   - `report.html`;
   - `diagnostics.html`.

Execution is sequential by design.

## 4. Configuration Architecture

The project has two config layers.

### 4.1 User-facing config (simple profile)

Default file: `config.yaml` (generated by `init-config`).

Contains only scenario-level fields:
- image folder;
- selected model labels;
- selected modes;
- optional runtime knobs (context/output limits, etc.).

### 4.2 Internal runtime config (full shape)

Canonical schema loaded into `BenchmarkConfig` (`src/config.py`).

Simple profile is expanded into full internal config by `expand_user_config()` (`src/user_config.py`) before validation and execution.

Mixed style is rejected explicitly.

## 5. Model Registry and Selection

Model discovery is registry-based.

### 5.1 Sources and generated files

- Raw LM Studio inventory: `models/lmstudio-models.raw.json`
- Active VLM candidates: `models/models.active.yaml`
- Excluded candidates with reasons: `models/models.excluded.yaml`
- User-facing label registry: `models.registry.yaml`

`refresh-models` regenerates these from live LM Studio `/api/v1/models`.

### 5.2 Selection rule

Active candidates are filtered by:
- valid model identifier extraction;
- LLM-like model type;
- vision capability.

There is no hard parameter-count cutoff in current architecture.
User controls final selection through `config.yaml` model label comments.

### 5.3 Reasoning profiles

Reasoning source of truth is registry field `reasoning` (`default` / `on` / `off`), not label suffix parsing.

## 6. LM Studio API Contracts

Default endpoints:
- model management: `http://localhost:1234/api/v1`
- OpenAI compatibility base: `http://localhost:1234/v1`

Primary transport:
- `POST /api/v1/chat`

Model lifecycle calls:
- `GET /api/v1/models`
- `POST /api/v1/models/load`
- `POST /api/v1/models/unload`

Runner keeps inference attached to loaded instance lifecycle and performs best-effort cleanup.

## 7. Prompting and Response Formats

Prompt builder: `src/prompts.py`.

Modes:
- `ru_free`, `en_free`
- `ru_pool`, `en_pool`
- `ru_pool_explained`, `en_pool_explained`

Primary response-format strategy is line-based in current configs:
- tags per line for free/plain pool modes;
- IDs per line for explained pool modes.

`strict_json` remains supported as a compatibility path.

## 8. Parsing and Validation Contracts

Response normalization (`src/validator.py`) produces:
- accepted tags / IDs;
- rejected tags / IDs;
- parse/schema flags;
- pool compliance diagnostics;
- reasoning-leak diagnostics;
- error typing.

Pool violations are benchmark signal, not automatic runtime failure.

## 9. Artifact Contracts

Run directory (`results/<run_id>/`) contains:
- `run_config.yaml` (resolved runtime config);
- `models.json`;
- `summary.csv`;
- `report.html`;
- `diagnostics.html`;
- `diagnostics.json`;
- `errors.log`;
- request artifacts under `requests/<request_id>/`.

`requests/<request_id>/...` is the source of truth for recomposition (`collect` + `report`).

## 10. Resume and Determinism

Runtime mode is controlled by config (`runtime.result_mode`):
- `deterministic`;
- `overwrite`;
- `accumulate`.

Resume logic avoids recomputing successful request artifacts in deterministic flow.

## 11. Reporting Layer

`src/report.py` builds static HTML.

Current matrix semantics:
- rows: image + mode;
- columns: models;
- chips: accepted/rejected/error markers;
- model header timing:
  - total latency;
  - optional per-pool averages for present pool modes.

`diagnostics.html` renders technical tables from `diagnostics.json`.

## 12. Test Strategy

Core suites:
- config + user config expansion;
- model registry;
- LM Studio client normalization behavior;
- runner flow and artifacts;
- report rendering;
- collect recomposition.

Tests are local-first and use fakes/fixtures for API-dependent paths where possible.

## 13. Stability Rules for Release

Before release, preserve these invariants:
- simple config workflow (`init-config -> dry-run -> run`);
- resolved config compatibility with `BenchmarkConfig`;
- registry-driven model label resolution;
- LM Studio REST Chat as primary inference path;
- reproducible artifact structure.
