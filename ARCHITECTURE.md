# Architecture

Technical contract for the Local VLM Image Tagger Benchmark.

## Purpose

The project is a local, CLI-first benchmark runner for vision-language models served by LM Studio.

It is intentionally small:

- single-process execution;
- sequential model runs;
- deterministic file artifacts;
- static HTML reports;
- no web backend or database.

Operational workflow belongs in [PROJECT_GUIDE.md](PROJECT_GUIDE.md).

## Non-Goals

Version 1 does not include:

- GUI;
- web server;
- SQLite or another persistent database;
- distributed or async execution;
- automatic quality judge model;
- plugin system;
- non-LM-Studio backends.

## Runtime Pipeline

Canonical `main.py run` path:

1. Load config (`src/config.py`).
2. Detect config style:
   - simple user profile;
   - full internal config.
3. Expand simple config with `src/user_config.py` when needed.
4. Validate semantic constraints (`src/validator.py`).
5. Load tag pools (`src/tag_pools.py`).
6. Discover input images (`src/image_loader.py`).
7. Create run directory and manifest artifacts (`src/storage.py`).
8. Connect to LM Studio (`src/lmstudio_client.py`).
9. For each selected model:
   - unload leftovers best-effort;
   - load model with configured load payload;
   - run optional image smoke request;
   - run every `(image, mode)` request;
   - save raw and normalized artifacts;
   - append `summary.csv`;
   - unload model when configured.
10. Build `report.html` and `diagnostics.html` (`src/report.py`).

## Configuration Layers

### Simple User Profile

Default file: `config.yaml`, generated by `init-config`.

Contains:

- input image folder;
- selected model labels;
- selected modes;
- optional runtime knobs.

LM Studio URLs are intentionally not exposed in this profile.

### Full Internal Config

Full shape is loaded into `BenchmarkConfig`.

Files under `configs/` are kept for smoke tests, examples, and advanced workflows. Simple and full config styles must not be mixed in one file.

## Model Registry

Model selection is registry-based.

Generated local files:

```text
models/lmstudio-models.raw.json
models/models.active.yaml
models/models.excluded.yaml
models.registry.yaml
```

`refresh-models` regenerates these from live LM Studio model inventory.

Active candidates require a usable model identifier and vision capability. The user chooses final labels in `config.yaml`.

Reasoning behavior is stored in the registry field `reasoning` (`default`, `on`, `off`). Code must not infer reasoning behavior from label suffixes.

## LM Studio API Contract

Default endpoints:

```text
http://localhost:1234/api/v1
http://localhost:1234/v1
```

Primary inference transport:

```text
POST /api/v1/chat
```

The benchmark sends prompt headers through the request-level `system_prompt` field. The `input` array contains only user-side content: the optional pool/tag list and the image.

Lifecycle calls:

```text
GET  /api/v1/models
POST /api/v1/models/load
POST /api/v1/models/unload
```

The runner keeps inference tied to the loaded model lifecycle and performs best-effort cleanup before and after model runs.

## Prompt and Response Contract

Prompt builder: `src/prompts.py`.

Modes:

```text
ru_free
ru_pool
ru_pool_explained
en_free
en_pool
en_pool_explained
```

Current default response formats:

- `line_tags` for free/plain pool modes;
- `line_ids` for explained pool modes.

`strict_json` remains supported as a parser compatibility path.

Mode prompt files are plain-text headers. They are not templated; braces remain literal text.

## Normalization Contract

Response normalization (`src/validator.py`) produces:

- accepted tags and IDs;
- rejected tags and IDs;
- parse/schema flags;
- pool compliance diagnostics;
- reasoning diagnostics;
- error typing.

REST reasoning content is diagnostic data. It must not be parsed as tags.

If a model writes reasoning-like prose into final answer content, the parser may mark `reasoning_leak_detected` and try to recover compact tag lines.

## Artifact Contract

Run directory:

```text
results/<run_id>/
  run_config.yaml
  models.json
  run_manifest.json
  run_state.json
  run_complete.json
  summary.csv
  report.html
  diagnostics.html
  diagnostics.json
  errors.log
  requests/
  raw/
  normalized/
  assets/thumbs/
```

`requests/<request_id>/...` is the canonical source for recomposition. `collect` rebuilds derived outputs from request artifacts.

## Resume and Result Modes

`runtime.result_mode` controls write behavior:

- `deterministic`: one canonical result per logical request;
- `overwrite`: recompute canonical files;
- `accumulate`: keep append-only attempts under `requests/<request_id>/attempts/`.

Resume skips successful deterministic request artifacts and may retry failed requests when configured.

## Reporting

`src/report.py` builds static HTML.

`report.html`:

- shows an answer matrix (`image x mode x model`);
- renders accepted, rejected, and error chips;
- links to `diagnostics.html`;
- shows model timing in column headers.

`diagnostics.html` renders technical tables from `diagnostics.json`.

## Test Coverage

Core test areas:

- config loading and simple config expansion;
- model registry generation and label resolution;
- LM Studio client request/response handling;
- runner flow and artifact writing;
- response parsing and pool validation;
- report rendering;
- collect recomposition.

## Stability Invariants

Before release, preserve:

- `init-config -> dry-run -> run` as the primary workflow;
- registry-driven model label selection;
- LM Studio REST Chat as the primary inference path;
- deterministic, rebuildable artifact structure;
- local-only runtime outputs.
