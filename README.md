# Local VLM Image Tagger Benchmark

Local CLI benchmark for comparing vision-language models in LM Studio on image tagging tasks.

Main workflow:

```text
ImgToTag/ -> python main.py init-config -> python main.py run --config config.yaml -> results/<run_id>/report.html
```

The runner loads selected models, sends image prompts through LM Studio REST Chat (`/api/v1/chat`), saves raw and normalized artifacts, writes `summary.csv`, and builds static HTML reports.

## Installation

### Required Software

- Python 3.10 or newer. Python 3.11+ is recommended.
- LM Studio with the local server enabled.
- At least one vision-language model downloaded in LM Studio.
- Git is optional. You only need it if you clone the repository instead of downloading a source archive.

NVIDIA drivers and `nvidia-smi` are optional. When available, the benchmark records GPU memory diagnostics. When unavailable, the run continues without GPU diagnostics.

### Get the Project

With Git:

```bash
git clone https://github.com/GooDgRaF/lmstudio-vlm-tag-benchmark.git
cd lmstudio-vlm-tag-benchmark
```

Or download the source archive, extract it, and open a terminal in the extracted project folder.

### Install Python Dependencies

Windows PowerShell:

```powershell
py -3 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
```

macOS/Linux:

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
```

Check that the CLI starts:

```bash
python main.py --help
```

### Prepare LM Studio

1. Install and open LM Studio.
2. Download at least one vision-language model.
3. Start the local server.
4. Keep the default server address: `localhost:1234`.

## Quick Start

1. Start the LM Studio server on `localhost:1234`.
2. Put images into `ImgToTag/`.
3. Generate the user config:

```bash
python main.py init-config
```

4. Edit `config.yaml`: keep the model labels and modes you want.
5. Validate the plan without inference:

```bash
python main.py dry-run --config config.yaml
```

6. Run the benchmark:

```bash
python main.py run --config config.yaml
```

Open:

```text
results/<run_id>/report.html
```

## Core Commands

```bash
python main.py init-config
python main.py refresh-models
python main.py list-models
python main.py validate-config --config config.yaml
python main.py dry-run --config config.yaml
python main.py run --config config.yaml
python main.py run --config config.yaml --run-id smoke-001
python main.py run --config config.yaml --run-id smoke-001 --force-lock
python main.py collect --run results/<run_id> --write-reports
python main.py report --run results/<run_id>
```

Advanced full-shape configs are available for development and checks:

```bash
python main.py dry-run --config configs/config.smoke.yaml
python main.py dry-run --config configs/config.example.yaml --limit 1
python main.py validate-config --config configs/config.rest-reasoning-smoke.yaml
```

Do not run `configs/config.example.yaml` without `--limit` unless you intentionally want a full model x image x mode benchmark.

## Configuration

The primary user-facing config is `config.yaml`, generated by `init-config`.

It contains:

- `images_folder`;
- selected `models`;
- selected `modes`;
- optional runtime knobs such as `limit_images`, `context_length`, and `max_output_tokens`.

LM Studio URLs are not exposed in the simple config. The defaults are:

```text
http://localhost:1234/api/v1
http://localhost:1234/v1
```

The project also supports full internal configs under `configs/` for smoke tests and advanced workflows.

## Modes

Supported modes:

- `ru_free`: Russian free tags;
- `ru_pool`: Russian tags from the plain pool;
- `ru_pool_explained`: Russian IDs from the explained pool;
- `en_free`: English free tags;
- `en_pool`: English tags from the plain pool;
- `en_pool_explained`: English IDs from the explained pool.

Current prompts use line-based responses:

- free/plain pool modes: one tag per line;
- explained pool modes: one ID per line.

`strict_json` parsing is still supported for alternative configs, but it is not the default path.

Prompt files under `prompts/` are plain-text mode headers. During REST calls, the header is sent as LM Studio `system_prompt`, while the pool/tag list is sent as user input together with the image.

There is no prompt templating. If you want different wording or limits, edit the prompt text directly.

## Reasoning Models

LM Studio REST responses may separate final answer content from reasoning content. The benchmark parses tags only from final answer content.

If a model writes reasoning-like text directly into the final answer, the parser marks this as a reasoning leak and tries to recover compact tag lines. The report shows a `thought anyway` marker when that happens.

Reasoning behavior is selected from the model registry field `reasoning` (`default`, `on`, `off`), not from label suffix parsing.

## Outputs

Each run creates:

```text
results/<run_id>/
  run_config.yaml
  models.json
  run_manifest.json
  run_state.json
  run_complete.json
  summary.csv
  report.html
  diagnostics.html
  diagnostics.json
  errors.log
  requests/
  raw/
  normalized/
  assets/thumbs/
```

Important files:

- `report.html`: human-facing answer matrix (`image x mode x model`);
- `diagnostics.html`: technical diagnostics;
- `summary.csv`: spreadsheet-friendly summary;
- `requests/<request_id>/...`: canonical per-request artifacts;
- `raw/`: raw LM Studio responses;
- `normalized/`: parsed responses and diagnostic flags;
- `errors.log`: runner events and errors.

`summary.csv`, `diagnostics.json`, `report.html`, and `diagnostics.html` can be rebuilt from request artifacts:

```bash
python main.py collect --run results/<run_id> --write-reports
```

## Repository Map

```text
main.py               CLI entrypoint
src/                  application code
tests/                pytest suite
configs/              full-shape configs
prompts/              prompt headers and tag pools
models/               generated local model inventory snapshots
results/              generated run artifacts
```

Generated local files such as `config.yaml`, `models.registry.yaml`, `models/*`, `results/*`, and `.tmp/` are ignored by git.
