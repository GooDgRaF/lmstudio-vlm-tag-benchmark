# SPEC-28 - Expand user run profile into internal `BenchmarkConfig`

## Goal

Support a simple user-facing run profile while preserving the current internal full config and runner behavior.

The user-facing file is intentionally not the full runtime config for the benchmark engine. It is a compact run profile edited by the user:

```yaml
images_folder: "ImgToTag"
limit_images: 1

models:
  - "qwen3-vl-4b-q4_k_m"

modes:
  - "ru_free"

output_folder: "results"
```

When the user runs:

```bash
python main.py run --config config.yaml
```

the program should first expand this user profile into the existing full internal config shape expected by `BenchmarkConfig`, then continue through the same runner path as a full config. In other words, `config.yaml` is the user's scenario file; the expanded full config is the real runtime contract.

## Two-level config semantics

The project should have two explicit config levels:

1. **User run profile**: short `config.yaml`, generated by `init-config`, edited by commenting/uncommenting models and modes.
2. **Resolved internal config**: full config dict produced automatically from the run profile and consumed by the existing runner.

Do not make the runner understand two separate runtime shapes. Keep one canonical internal shape: `BenchmarkConfig`.

For traceability, runs started from a user profile should save both layers when practical:

```text
results/<run_id>/config.user.yaml       # exact user-edited profile
results/<run_id>/config.resolved.yaml   # expanded full internal config
```

If only one artifact is practical in the first pass, `run_config.yaml` must contain the resolved full config, not the short profile.

## Full config support

Existing full configs must continue to work unchanged:

```text
config.smoke.yaml
config.example.yaml
```

Detection rule:

- if YAML root has full-config sections such as `lmstudio`, `input`, `output`, `generation`, `load`, or `runtime`, treat it as the current full config;
- if YAML root has `images_folder` and `models` is a list of strings, treat it as a user run profile;
- if a file mixes both styles, fail with a clear error instead of guessing.

Example mixed-style error:

```text
Config mixes user profile and full internal config fields.
Use either generated `config.yaml` style or full `config.example.yaml` style, not both.
```

Do not break current tests that use full config builders.

## New module

Add:

```text
src/user_config.py
```

Responsibilities:

- parse simple `config.yaml`;
- validate simple fields;
- load `models.registry.yaml`;
- resolve model labels to full model entries;
- build a full internal config dict;
- optionally save the resolved config into the run directory later.

## Simple config schema

Supported keys:

```yaml
images_folder: str                    # required
limit_images: int | null              # optional, default null or generated as 1
models: list[str]                     # required, labels from registry
modes: list[str]                      # required
output_folder: str                    # optional, default "results"
context_length: int                   # optional, default 8192
max_output_tokens: int                # optional, default 4096
temperature: float                    # optional, default 0.0
recursive: bool                       # optional, default false
```

Unknown top-level keys should produce a clear warning or error. Prefer error for first implementation because typo detection matters.

Example:

```text
Unknown simple config key: max_tokens
Did you mean: max_output_tokens?
```

## Mode validation

Allowed modes:

```python
ALLOWED_MODES = [
    "ru_free",
    "ru_pool",
    "ru_pool_explained",
    "en_free",
    "en_pool",
    "en_pool_explained",
]
```

Reject unknown modes with a clear error.

## Model resolution

For simple configs:

```yaml
models:
  - "qwen3_5-9b-q4_k_m-no-think"
```

Resolve each label through `models.registry.yaml` into a full model entry:

```yaml
- id: "qwen/qwen3.5-9b@q4_k_m"
  base_model_id: "qwen/qwen3.5-9b"
  label: "qwen3_5-9b-q4_k_m-no-think"
  reasoning: "off"
  display_name: "Qwen3.5 9B"
  params: "9B"
  architecture: "qwen35"
  quant: "Q4_K_M"
  quant_bits: 4
  size_bytes: 6548927017
  max_context_length: 262144
```

The label suffix is not parsed during config expansion. Use the registry field `reasoning`.

If registry is missing:

```text
models.registry.yaml not found. Run `python main.py init-config` first.
```

If a label is missing:

```text
Unknown model label: ...
Run `python main.py init-config --force` to refresh available models.
```

## Integration with `src/config.py`

Refactor carefully.

Recommended approach:

1. Keep current dataclasses unchanged.
2. Split current `load_config(path)` logic into:
   - `load_config(path)` — public entry point;
   - `_load_full_config_from_mapping(loaded, cfg_path)` — current parser logic.
3. Add detection:

```python
def load_config(config_path):
    loaded = read_yaml(config_path)
    if is_simple_user_config(loaded):
        loaded = expand_simple_user_config(loaded, config_path)
    return _load_full_config_from_mapping(loaded, config_path)
```

This lets old full configs and new simple configs share the same validation and runner.

## Resolved config artifact

After expansion, the full resolved config should be available in memory as `cfg.raw`.

The runner already writes `run_config.yaml` in results in current versions. If it does, ensure that for simple configs it writes the resolved full config, not only the short user config.

Optional but recommended:

```text
results/<run_id>/config.user.yaml       # exact user config snapshot
results/<run_id>/config.resolved.yaml   # expanded internal config
```

If this is too much for the first implementation, at least keep current `run_config.yaml` as the resolved config.

## Acceptance criteria

- Simple `config.yaml` loads successfully.
- Full existing configs still load successfully.
- Simple model labels resolve through the registry.
- `max_output_tokens` maps to internal `generation.max_tokens`.
- `context_length` maps to internal `load.context_length`.
- `limit_images` maps to internal `limits.limit_images`.
- Running `dry-run --config config.yaml` works after `init-config`.
- Running `run --config config.yaml` reaches the same runner path as full configs.
