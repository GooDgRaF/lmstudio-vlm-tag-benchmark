# SPEC-30 - Tests, docs, and migration cleanup

## Goal

Update tests and documentation so the new user-friendly config becomes the primary documented workflow, while current full configs remain available as advanced/internal examples.

## Documentation updates

Update at least:

```text
README.md
PROJECT_GUIDE.md
ARCHITECTURE.md
AGENTS.md
```

If the repository has a `specs/` directory, add links there too.

## README changes

The quick start should become:

```bash
python main.py init-config
python main.py dry-run --config config.yaml
python main.py run --config config.yaml
```

Explain:

- `init-config` queries LM Studio;
- `config.yaml` is the file the user edits;
- model labels and modes are already listed inside their corresponding fields;
- user only comments/uncomments list items;
- `config.example.yaml` is advanced/internal and not the main entry point.

## PROJECT_GUIDE changes

Add a section:

```text
Human-friendly config workflow
```

Include:

```text
config.yaml              user-edited simple config
models.registry.yaml     generated model label registry
models/*.yaml/json       generated LM Studio model inventory
config.example.yaml      full internal/advanced example
config.smoke.yaml        legacy smoke/full-shape config
```

Mention that simple config is expanded into the current `BenchmarkConfig` internally.

## ARCHITECTURE changes

Update the config architecture section.

Required statement:

```text
The project has two config levels:
1. human-friendly `config.yaml`, edited by the user;
2. full internal config dict, produced by expansion and consumed by the existing runner.
```

Clarify that LM Studio endpoints are no longer user-facing settings in the simple config. They remain defaults in code:

```text
http://localhost:1234/api/v1
http://localhost:1234/v1
```

Clarify that the main inference transport remains:

```text
POST /api/v1/chat
```

## AGENTS changes

Add implementation guidance:

- Do not remove support for full configs.
- Do not make users copy model labels from a reference section; labels must be directly under `models:`.
- Do not put LM Studio URL into the generated simple config.
- Keep defaults `context_length = 8192` and `max_output_tokens = 4096`.
- Keep reasoning source of truth in model registry field `reasoning`, not label parsing.

## Migration notes

Current full configs can remain:

```text
config.smoke.yaml
config.example.yaml
```

Do not delete them during this feature unless a later cleanup spec explicitly says so.

Generated new files:

```text
config.yaml
models.registry.yaml
models/lmstudio-models.raw.json
models/models.active.yaml
models/models.excluded.yaml
```

Consider adding these to `.gitignore` if they are machine/user-specific. If there is no `.gitignore`, add one or update docs to say generated files may be local.

Recommended `.gitignore` entries:

```gitignore
config.yaml
models.registry.yaml
results/
```

Do not ignore `models/models.active.yaml`, `models/models.excluded.yaml`, or `models/lmstudio-models.raw.json` unless the implementation decides they should be purely local. The spec requires them to be auto-generated by `refresh-models`, but they may remain versioned as a useful project snapshot if that matches the current repository convention.

Remove stale `list-models --config ...` documentation and examples from:

- `README.md`
- `PROJECT_GUIDE.md`
- `ARCHITECTURE.md`
- `AGENTS.md`
- archived or active spec references only when they are part of current workflow documentation.

But if the project wants to version an example registry for tests, use:

```text
tests/fixtures/models.registry.yaml
```

not the user's generated root registry.

```bash

python main.py run --config config.yaml --run-id simple-config-smoke
```
